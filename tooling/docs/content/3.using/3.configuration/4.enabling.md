# Pourquoi devez-vous activer une ressource ?

Par défaut, le magasin ne matérialisera rien. Vous devez permettre aux modèles que vous voulez de se matérialiser. Nous fonctionnons de cette façon pour éviter de matérialiser des données dont vous n'avez pas besoin: si un seul tableau de bord du magasin vous intéresse, vous n'avez pas à matérialiser tout le magasin.

# Comment activer une ressource ?
> Une ressource devrait être comprise comme étant un ensemble de tables requises par un tableau de bord, comme un comptoir entier ou comme un unique fichier `.sql`.

::alert{type="info"}
L'inscription `+enabled: True|False` est un mécanisme DBT pour activer ou désactiver des ressources. Vous devriez lire la [documentation DBT](https://docs.getdbt.com/reference/resource-configs/enabled) pour en apprendre davantage à ce sujet. Idem pour **tests** et **seeds**. 
::

Activer un modèle se fait par le fichier `cssXX.dashboards_store/dbt_project.yml`. Pour activer un modèle, il faut simplement ouvrir le fichier `dbt_project.yml`, trouver le nom de la ressource que vous voulez activer et changer la note `+enabled: false` à `+enabled: true`. 

## Exemple

### Activation d'un comptoir

J'ai ajouté mes données statiques au comptoir des ressources humaines et je veux maintenant l'activer. 

1. Je commence par ouvrir mon `cssXX.dashboards_data/dbt_project.yml` et je cherche la ressource `human_resources`. Je trouve la chose suivante :

```yaml
models:
    store:
        marts:
            human_resources:
                +enabled: False 
```

2. Je bascule de `+enabled: False` vers `+enabled: True` :

```yaml
models:
    store:
        marts:
            human_resources:
                +enabled: True
```

3. ...et c'est fait!

### Activation d'un tableau de bord

Disons que je veuille activer le tableau de bord `effectif_css` dashboard.

1. Je commence par ouvrir mon `cssXX.dashboards_data/dbt_project.yml` et je cherche pour la ressource `effectif_css`. Je trouve la chose suivante :

```yaml
models:
    store:
        dashboards:
            effectif_css:
                +enabled: False
```

2. Je bascule de `+enabled: False` vers `+enabled: True` :

```yaml
models:
    store:
        dashboards:
            effectif_css:
                +enabled: True
```


### Activation d'une interface
::alert{type="info"}
Pour activer une interface, la base de données doit être liée. Référez-vous à la section [activation d'une base de données](/using/configuration/linking) pour plus d'information.
::

Disons que je doive activer l'interface `paie` pour matérialiser le comptoir RH.

1. J'ouvre d'abord mon `cssXX.dashboards_data/dbt_project.yml` et je cherche la ressource `paie`. Je trouve la chose suivante: 

```yaml
models:
    store:
        interfaces:
            paie:
                +enabled: False
vars:
    database_paie: "replaceMeWithYourDatabase" 
```

2. Je remplace ensuite la variable `database_paie` avec le nom de ma base de données (se référer à la section [activation d'une base de données](/using/configuration/linking) pour plus d'information) :

```yaml
vars:
    database_paie: "dataServer.paie"  # Le nom de la base de données dans votre entrepôt. Il sera interpolé dans la couche Interface.
```

3. Je bascule ensuite de `+enabled: False` vers`+enabled: True` :

```yaml
models:
    store:
        interfaces:
            paie:
                +enabled: True
vars:
    database_paie: "dataServer.paie"  
```

## Entre les dépendances de ressources

Une erreur commune qui est faite est d'activer un modèle sans avoir activé ses dépendances :
* Pour les **comptoirs**, cela signifie d'activer un comptoir sans avoir activé les `interfaces` desquelles il dépend.
* Pour un **tableau de bord**, cela signifie d'activer un tableau de bord sans avoir activé les comptoirs à partir desquels il est construit.

::alert{type="warning"}
On retrouve habituellement dans les pages des tableaux de bord et des comptoirs la liste de leurs dépendances. En activant une telle ressource, assurez-vous d'avoir aussi activé ses dépendances en amont.  
::
